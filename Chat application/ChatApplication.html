My Chat Application Journey

Visual Studio 2019, C++, .NET, Visual C++

Chapter 1: Getting Started

With my eyes set upon a project, I began with plain-old research. I had no expirience with networking, so lots of reading and learning had to be done! Once I became familiar with the idea of sockets, I needed to choose a library to use for my project (that also was compatable with CPP). This is when I came across Winsock. "Winsock enables programmers to create advanced Internet, intranet, and other network-capable applications to transmit application data across the wire, independent of the network protocol being used."

I followed the winsock tutorial on how to set up a basic synchronous echo server. Along with the Winsock library, I used the following websites to also help me understand sockets and other concepts.

  • Ncona
  • GeeksforGeeks
  • BogoToBogo

After following the Winsock guide, I had come up with a simple server for one client. The message echoed was hard-coded into the client. However, the message echoed back did not come as expected; I had only received (echoed) the number of bytes back, rather than the message "This is a testppppp." I was glad information was able to be send and echoed, but there was still lots of work to be done.

Chapter 2: Recieving and Sending Proper Messages

After reviewing the code and adding some simple changes (which resulted in the bytes being decoded), I received the desired message...almost. There was (what semed to be) ASCII garbage following the message echoed.

After doing extensive research on the nature of sending and receiving using Winsock, I learned that the empty characters in the message (buffer) are filled with random ASCII characters. Think of it as an Amazon box with an item, and the empty space in the box is filled with bubblewrap. I fixed this issue using the function memset(). This function replaces all the characters after (the null terminating character) '/0' with NULL characters.

I then added some foundation such as a welcome message, and partial user interface where the user can type the message to send (rather than hardcoded).

Chapter 3: Introducing a Cycle

I learned more about chat applications and networking through LuminousMen.com and What-When-How.com. I learned about the differences between synchronous and asynchronous networking (I/O). My current chat application was synchronous, as in specific queues (sending/receiving) determined the cycles of input and output. That being said, you cannot have a client send a message while it is trying to receive one*. However, with neat tricks, you can get around this.

Using Sloan Kelly's video as a basis and new starting point for my server code, I began integrating integrating synchronous I/O with select().

Initially with one client, the server would (after the welcome message) wait to receive, then send the echo back to the client, which becomes a problem with multiple clients, as mentioned in the paragraph above. However, Mr. Kelly utliizes loops with select() along with the struct fd_set{} type. This data type holds member function of an array of sockets (fd_array[]). We then loop through this everytime a message was recieved (to the server) to determine who (clients) to forward the message to (when we have multiple clients, we don't want to send client's message back to themself).

This specific video was a tremendous help with not only me grasping the specifics of networking and sockets between a server and client, but also with a basis for my code. I referenced a portion of my server code from this video. This allowed me to build upon it and later integrate multithreading.

Once I implemented this new server code, bugs of course arose. The first I encounted was when one client was connected. I was only able to send one message.

As referenced above (in bold) we do not want to send a client's message back to itself. To do this, we look at the different sockets in play. We loop the socket array (from the fd_array[]) and if the socket is not the server(listening) socket or the client socket, we send the message to that socket (for it is a different client).

Now the client will talk to itself(nothing echoed) if it is the only one connected. otherwise, its message is sent to the other clients.

Chapter 4: Proper Conversations via Multithreading

Another major bug I had experienced at this stage dealt with mutliple clients. I experienced a problem where messages would not initially go through once sent from a client. To receive a message from a client, a different client had to first send a message. (You would only be able to see a pervious message send after the next client had sent a message.

I soon figured out that this is because each client that is supposed to receive the message has their loop attempting to send a message before receiving one. This again partially alludes to the synchrounous behavior of the server/client.

After lots of reading a came across a possible solution: multithreading. However, my runtime compiler, MinGW, does not support threads. After lots of searching, I came across a Github user Meganz. He/she had developed custom libraries to import so that threading would compile using MinGW.

Now that I had the tools at my disposal, I needed to do lots of testing and learning for I had no expirience with threading at the time. I used a simple program found on StackOverflow as a starting point. I fidgited around with this and slowly transformed the code and integrated it into my client code. One interesting part of threading I came across was the fact of special parameters with the function I used, _beginthreadx()

After much of reading and testing, I finally was able to incorporate multi-threading. The thread checks for incoming messages by constantly attempting to receive messages from the server. I incorporated a sleep() on the receiving thread to limit CPU usage. Now, clients can send as many messages as they like, while still receiving messages from the server in real* time.

Chapter 5: Basic Security

The next step I wanted to embark upon was security. I wanted to track and maintain user's connections and store them in a database. I needed to store a key(ip address in this case) as well as a key-value(the number of connections per IP in this case). The std::map was perfect for this for "Maps are associative containers that store elements formed by a combination of a key value and a mapped value, following a specific order."

I utilized the function getpeername() to extracted the IP from a client's socket.

I then implemented incrementing and decrementing on the server side to maintain the total number of connections. Further. I applied a threshold of maximum clients per IP address to prevent fork-bombing/overloading. If a client tries to join from a network that has already reach the maximum capacity (from their specific IP), they are met with an informative message, and the client closes automatically after some time.

This process took more time than I thought, and I had to decrypt and solve a good number of logic bugs. This included everything from correctly closing a socket/adding the (correct) client socket to the server, adding/erasing the IP objects from the map, and incrementing/decrement its numConnected integer. For the graphical user interface, I used Visual studio's UI templates. I started out with a CLR Empty Project based off of .NET Framework.

Began learning more and more CLR Framework and window forms. I soon acheived a basic graphical interface that I could structure future edits around. However, I still needed to actually integrate my code into this GUI. (16).

Now that I had a basic structure to work around, I began transfering my code in this new, .NET environment. However, after minor confusion and a pleathora of errors, I came across the fact that .NET does not support traditional threading, for the CLR framework had its own integration of threads. I altered my mapping in reference to < a href = " https://docs.microsoft.com/en-us/cpp/dotnet/map-stl-clr?view=msvc-160"> System::String

I needed to integrate a new type of threading: one that is compatible with C++/CLI applications. I learned more about threading in CLI applications on Codeproject's website.

After much research, I decided to use C++/CLI's BackgroundWorker Class. This allows for operation executions on separate threads. After implementing and testing the backgroundworker class into my code, I was able to succesfully communicate (I/O) from my Client Window with another (consol) client through a (consol) server.

After acheiving a sound baseline with my client window, I moved on to the server window. I displayed usefull data to the server regarding booting updates, connections, messages, and IP updates.
